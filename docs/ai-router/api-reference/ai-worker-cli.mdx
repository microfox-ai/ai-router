## Overview

`@microfox/ai-worker-cli` provides command-line tools for running workers locally, compiling them into a deployable serverless build, and scaffolding new worker files. It scans your codebase for worker definitions, builds Lambda handlers, and generates Serverless Framework configuration.

**Deployment is handled by the `microfox` CLI** (`npx microfox deploy` / `npx microfox push`) â€” the `ai-worker` CLI never deploys. See [Deploying with the microfox CLI](#deploying-with-the-microfox-cli) below.

## Installation

```bash
npm install -D @microfox/ai-worker-cli
# or use npx
npx @microfox/ai-worker-cli@latest <command>
```

## Invocation

You can run the CLI in any of these ways:

- **Direct**: `npx @microfox/ai-worker-cli <command>`
- **Short alias**: `npx ai-worker <command>` (e.g. `npx ai-worker dev`, `npx ai-worker compile`, `npx ai-worker new`)

**Typical workflow:**

```bash
npx ai-worker new                # 1. scaffold a worker or queue in app/ai/
npx ai-worker boilerplate        # 2. add the Next.js API routes + job stores (once per app)
npx ai-worker dev                # 3. run + test workers locally (no AWS needed)
npx microfox@latest deploy       # 4. compile + push to the Microfox platform
```

## Commands

### `dev` (local dev server)

Runs your workers locally with **zero AWS and zero Microfox account**: one HTTP server exposing the same routes as a deployed core group, workers invoked in-process, an in-memory queue standing in for SQS, and hot reload.

```bash
npx ai-worker dev
# custom port
npx ai-worker dev --port 4200
```

**Options:**

- `-p, --port <port>` - HTTP port (default: `4100`)
- `--ai-path <path>` - Directory scanned for `*.worker.ts` / `*.queue.ts` files (default: `app/ai`)
- `-c, --concurrency <n>` - Max concurrent runs per worker (default: `5`)
- `--max-receive-count <n>` - Delivery attempts before a message lands in the local DLQ (default: `3`)

**Routes** â€” the same routes as a deployed core group, with the same request/response shapes:

- `POST /workers/trigger` - trigger a worker
- `GET /workers/config` - worker registry/config
- `GET /docs.json` - generated docs
- `POST /queues/{id}/start` - start a queue

Point your app at it by setting:

```bash
WORKER_BASE_URL=http://localhost:4100
```

The auth rule matches production: endpoints are open unless `WORKERS_API_KEY` is set â€” then the `x-workers-trigger-key` header is required.

**Dev-only extras:**

- `GET /jobs/{jobId}` - job run tree + queue doc for one job
- `GET /jobs` - all local jobs
- `GET /dlq` - locally dead-lettered messages
- `GET /health` - health check

**Environment:** loads `.env` â†’ `.env.local` â†’ `.env.dev` â†’ `.env.dev.local` (later files win; real shell env wins over all files) and forces `ENVIRONMENT`/`STAGE`/`NODE_ENV` to `dev`.

**Job store:** uses a local file-persisted store (`.microfox/dev-state.json`, auto-gitignored) when no Upstash/Mongo env is present. If `UPSTASH_*`/`MONGODB_URI` env exists (or `WORKER_DATABASE_TYPE` is set), the real store is used instead. Parked HITL approvals and job history survive dev-server restarts via the state file.

**Using your app's UI against the local store:** set `WORKER_DATABASE_TYPE=local` for **both** the dev server and your Next.js app (plus `WORKER_BASE_URL=http://localhost:4100`). The boilerplate's `stores/localDevAdapter.ts` then proxies all app-side job/queue store reads *and* writes to the dev server's `/dev-store/*` API, so trigger â†’ poll â†’ HITL approve/reject all work in the browser with no Redis or Mongo. The `/dev-store` routes are only served when the dev server itself runs on the local store, and follow the same `WORKERS_API_KEY` auth rule. âš ď¸Ź `WORKER_DATABASE_TYPE=local` is dev-only â€” never deploy with it (compile prints a loud warning if it would ship into `env.json`).

**Hot reload:** edit any worker/queue/lib file while the server runs â€” the next invocation uses the new code, no restart needed. Adding or removing `*.worker.ts`/`*.queue.ts` files is picked up live. Type `rs` + Enter to force a re-scan.

**Differences from AWS (fidelity limits):**

- Queue schedules are printed at startup but **not executed** locally â€” start scheduled queues manually via `POST /queues/{id}/start`.
- All workers share one Node event loop (CPU-bound work blocks other runs) and one `process.env`.
- Delivery is exactly-once (SQS is at-least-once).
- A handler error recorded in the job store is **not redelivered** (same effective behavior as prod idempotency) â€” the local DLQ only receives failures that never reached the store (e.g. module-load/syntax errors).

### `compile`

Compiles workers into a deployable build in `.serverless-workers/`. **It does not deploy** â€” deploy the build with `npx microfox push` (or use `npx microfox deploy` to compile + push in one step).

```bash
npx @microfox/ai-worker-cli compile [group] [options]
# or
npx ai-worker compile [group] [options]
```

<Note>
`compile` was previously called `push` (which built **and** deployed). `push` still exists as a hidden, deprecated alias that only compiles and never deploys; its old flags (`--build-only`, `--deploy-only`, `--skip-deploy`, `--skip-core`) are gone or ignored. Use `compile` + `microfox deploy` instead.
</Note>

**Arguments:**

- `[group]` - Optional. Compile only this group (multi-group layout, e.g. `core`, `default`, `workflows`)

**Options:**

- `-s, --stage <stage>` - Deployment stage baked into the build (default: `STAGE` env var, then `prod`). Stage names must match `[a-z0-9-]`, max 20 characters. See [Multi-stage deploys](#multi-stage-deploys).
- `-r, --region <region>` - AWS region baked into the generated serverless config (default: `us-east-1`)
- `--ai-path <path>` - Path to the AI directory containing workers (default: `app/ai`)
- `--service-name <name>` - Override serverless service name (default: `ai-router-workers-<stage>`)
- `--skip-group <groups>` - Comma-separated groups to skip building (multi-group layout, e.g. `"core,workflows"`)
- `--require-auth` - Fail the build if no worker secret (`WORKERS_API_KEY` / projectId) can be resolved
- `--allow-public` - Build worker endpoints as public without a warning when no secret is set

**What it does:**

1. **Scans** for worker files (`*.worker.ts`) and queue files (`*.queue.ts`)
2. **Reads** optional `worker` and `env` config from `microfox.json` / `microfox.config.ts` (externalDeps, includeNodeModules, groups, env shipping rules)
3. **Extracts** worker configurations (`workerConfig` exports, including `group`)
4. **Analyzes** dependencies and environment variable usage
5. **Detects** worker-to-worker calls (`ctx.dispatchWorker`)
6. **Generates** Lambda handler files (one Serverless stack per group; core stack for trigger/config/queues)
7. **Generates** `serverless.yml` configuration and `env.json` (see [Controlling shipped env vars](#controlling-shipped-env-vars-env-block))
8. **Builds** handlers with esbuild (per-group options from `worker.groups`)

**Output:**

- `.serverless-workers/` - single-group build (serverless.yml + bundled handlers)
- `.serverless-workers/<group>/` - per-group builds when workers declare multiple groups

See [Worker Architecture â€“ Multi-group and Project config](../advanced/worker-architecture#multi-group-deployments) for groups and `microfox.json` details.

**Example:**

```bash
# Compile everything (stage defaults to prod)
npx ai-worker compile

# Compile for a specific stage
npx ai-worker compile --stage staging

# Compile only the "core" build
npx ai-worker compile core

# Compile everything except core
npx ai-worker compile --skip-group core
```

### `new`

Scaffolds a new **worker** (`.worker.ts`) or **queue** (`.queue.ts`). When run without `--type`, the CLI asks you to choose one option, then prompts for the ID if not provided.

```bash
npx @microfox/ai-worker-cli new [id] [options]
# or
npx ai-worker new [id] [options]
```

**Arguments:**

- `id` - Optional. Worker or queue ID (filename and config id). If omitted, the CLI prompts for it.

**Options:**

- `--type <worker|queue>` - Scaffold type. If set, skips the interactive "worker or queue?" prompt.
- `--dir <path>` - Output directory (workers: default `app/ai/workers`; queues: default `app/ai/queues`)
- `--schedule <expression>` - Optional schedule, workers only (e.g. `"cron(0 3 * * ? *)"` or `"rate(1 hour)"`)
- `--timeout <seconds>` - Lambda timeout, workers only (default: `300`)
- `--memory <mb>` - Lambda memory, workers only (default: `512`)

**Examples:**

```bash
# Interactive: choose worker or queue, then enter id
npx ai-worker new

# Create a worker (non-interactive)
npx ai-worker new video-processor
npx ai-worker new daily-report --schedule "cron(0 3 * * ? *)" --timeout 600

# Create a queue (non-interactive)
npx ai-worker new cost-usage --type queue

# Custom directory
npx ai-worker new data-processor --dir app/workers
```

**Generated worker file** (e.g. `app/ai/workers/video-processor.worker.ts`):

- Uses `createWorker` with Zod input/output, `ctx.jobStore`, `ctx.logger`, and `ctx.dispatchWorker`. Fill in input/output schemas and handler logic.

**Generated queue file** (e.g. `app/ai/queues/cost-usage.queue.ts`):

- Uses `defineWorkerQueue` with `id` and a `steps` array. Includes commented-out examples for `chain` (input mapping), `requiresApproval`/`resume` (HITL), and `delaySeconds`. Replace `first-worker` with your real worker IDs, test with `npx ai-worker dev`, and deploy with `npx microfox@latest deploy`.

### `boilerplate`

Creates or updates worker boilerplate files (job store, API routes, config) in your application.

```bash
npx @microfox/ai-worker-cli boilerplate [options]
# or
npx ai-worker boilerplate [options]
```

**Options:**

- `--force` - Overwrite existing files (default: skip existing files)
- `--app-dir <path>` - App directory path (default: `app`)
- `--skip-config` - Skip microfox.config.ts updates

**What it creates:**

1. **Auth helper** (`app/api/workflows/auth.ts`):
   - `getClientId(req)` stub â€” replace with your actual session/JWT logic to attach a userId to every worker job

2. **Job store adapters** (`app/api/workflows/stores/`):
   - `jobStore.ts` - Main job store interface (adapter selection based on `WORKER_DATABASE_TYPE`)
   - `mongoAdapter.ts` - MongoDB adapter
   - `redisAdapter.ts` - Upstash Redis adapter
   - `queueJobStore.ts` - Queue job tracking

3. **Workers HTTP API** (`app/api/workflows/workers/[...slug]/route.ts`):
   - POST `/workers/:workerId` â€“ trigger a worker (calls trigger API; no registry required)
   - GET `/workers/:workerId/:jobId` â€“ job status
   - POST `/workers/:workerId/webhook` â€“ webhook callback
   - POST `/workers/:workerId/update` â€“ job update (called by Lambda)
   - POST `/workers/:workerId/job` â€“ create job record before polling

4. **Queues API** (`app/api/workflows/queues/[...slug]/route.ts`):
   - POST `/queues/:queueId` â€“ trigger a queue (calls queue-start API; no registry required; pass first-worker input)
   - GET `/queues/:queueId/:jobId` â€“ get queue job status
   - GET `/queues` or GET `/queues?queueId=...` â€“ list queue jobs
   - POST `/queues/:queueId/update` â€“ update queue job step (called by Lambda)
   - POST `/queues/:queueId/webhook` â€“ webhook for queue completion
   - POST `/queues/:queueId/approve` â€“ HITL approve/reject a paused step (see [HITL Queues](../advanced/hitl-queues))

5. **Configuration** (`microfox.config.ts`):
   - Merges `workflowSettings.jobStore` block (if file exists)
   - Creates minimal config file if missing

**Example:**

```bash
# Create boilerplate files
npx ai-worker boilerplate

# Overwrite existing files
npx ai-worker boilerplate --force

# Use custom app directory
npx ai-worker boilerplate --app-dir src/app

# Skip config updates
npx ai-worker boilerplate --skip-config
```

For a runnable example that includes repository references, see [Workers + Orchestration Proof](../examples/workers-orchestration-proof).

## Deploying with the microfox CLI

Deployment lives in the `microfox` CLI, which uploads your compiled build to the Microfox platform â€” the platform runs `serverless deploy` for you, so **you do not need AWS credentials locally**. It requires a `microfox.json` (or `microfox.config.ts` with `workflowSettings.deploymentConfig`) containing a `projectId`, and authentication via `npx microfox login` (or a `MICROFOX_TOKEN` env var in CI).

### `microfox deploy`

Compile + push in one step (equivalent to `microfox compile` then `microfox push`). This is the everyday command â€” use it for first deploys and for every rebuild/redeploy after a change.

```bash
npx microfox@latest deploy [groupname] [options]
```

**Options:**

- `[groupname]` - Compile and push only this group (e.g. `core`, `default`, `workflows`)
- `--skip-group <groups>` - Comma-separated list of groups to skip (e.g. `"core,workflows"`)
- `-s, --stage <stage>` - Stage baked into the generated serverless config (see [Multi-stage deploys](#multi-stage-deploys))
- `-r, --region <region>` - AWS region baked into the generated serverless config
- `--ai-path <path>` - Path to the AI directory containing workers (default: `app/ai`)
- `--skip-compile` - Push the existing `.serverless-workers` build without recompiling

**Examples:**

```bash
npx microfox@latest deploy                      # compile + push all groups
npx microfox@latest deploy core                 # compile + push only the "core" group
npx microfox@latest deploy --skip-group core    # compile + push everything except core
npx microfox@latest deploy --skip-compile       # push the last build without recompiling
```

### `microfox compile` and `microfox push`

The two halves of `deploy`, when you want them separately:

- `npx microfox compile [group] [--skip-group ...] [--stage ...]` - build only (runs `ai-worker compile` under the hood, writes to `.serverless-workers/`, nothing is uploaded)
- `npx microfox push [group] [--skip-group ...] [--stage ...]` - upload only (bundles the existing `.serverless-workers/` build as a ZIP and uploads it to the Microfox platform for deployment; per group when using the per-group layout)

If `--stage` is omitted on `push`, the stage the build was compiled with is used (read from the build's `env.json`), defaulting to `prod`. Passing a `--stage` that differs from the compiled build prints a mismatch warning â€” recompile with that stage to avoid a mixed deploy.

### Multi-stage deploys

Both CLIs support deployment **stages**, so you can run `staging` and `prod` (or any stage) side by side:

```bash
# Build + deploy a staging stack alongside prod
npx microfox@latest deploy --stage staging

# Or as separate steps
npx ai-worker compile --stage staging
npx microfox@latest push --stage staging
```

- The stage is **baked into the build** at compile time (via `--stage` or the `STAGE` env var; default `prod`): `env.json`'s `ENVIRONMENT`/`STAGE`/`NODE_ENV`, the `serverless.yml` stage fallback, and SQS queue names all carry it, so each stage deploys as a **parallel CloudFormation stack**.
- Stage names must match `[a-z0-9-]`, max 20 characters.
- Stage-scoped env files load automatically: `.env` â†’ `.env.local` â†’ `.env.{stage}` â†’ `.env.{stage}.local` (later wins). See [Environment files](#environment-files-env-cascade).

<Warning>
Platform-side (console) support for non-prod stages is still rolling out â€” until it lands, non-prod deployments may appear as prod in the Microfox console.
</Warning>

### Monitoring deployments (`status`, `logs`, `metrics`)

Every `microfox push`/`deploy` returns a **Deployment ID**, which is also saved locally so `latest` resolves to your most recent deployment(s):

```bash
npx microfox status latest          # status of the most recent deployment
npx microfox status latest -n 3     # the 3 most recent (e.g. one per group)
npx microfox status <deploymentId>  # a specific deployment

npx microfox logs latest            # deployment logs
npx microfox logs latest --step compiling     # one phase: compiling | deploying | post_deployment
npx microfox logs <id> --limit 200 --cursor <iso-timestamp>

npx microfox metrics <deploymentId> # timing, CPU/memory, image size, data transfer
```

- **`status`** shows the deployment's state, group, error message (if any), start/end time, and â€” once deployed â€” the **Base URL** of your workers API. Point `WORKER_BASE_URL` at that URL.
- **`logs`** streams the platform-side build logs in three phases: `compiling`, `deploying`, and `post_deployment`. This is where to look when a deployment fails.
- **`metrics`** shows deployment timing (per step), resource usage (CPU/memory peaks), and network transfer.

The same information is available in the **Microfox console**: each deployment appears under your project with its status, logs, and the deployed endpoints. The CLI commands and the console read the same records â€” use whichever is closer at hand. For runtime (post-deploy) behavior, worker runs are tracked in your **job store** (see [Worker Architecture â€“ Monitoring](../advanced/worker-architecture#monitoring-and-observability)); each job records status, progress, output, and errors, queryable via `GET /api/workflows/workers/:workerId/:jobId`.

## Environment files (.env cascade)

`compile` (and `microfox compile`/`deploy`) reads env vars from a **stage-scoped cascade** of env files, dotenv-flow style â€” later files win on key conflicts:

```
.env  â†’  .env.local  â†’  .env.{stage}  â†’  .env.{stage}.local
```

- The `--stage` flag (or `STAGE` env var) decides which stage files load â€” e.g. `--stage staging` loads `.env` â†’ `.env.local` â†’ `.env.staging` â†’ `.env.staging.local`.
- Real shell/CI environment variables always win over all files.
- `ai-worker dev` uses the same cascade with the stage fixed to `dev` (`.env.dev`, `.env.dev.local`).

### Stage isolation (`files: 'isolated'`)

If you'd rather have each stage declare its **complete** env (no inheritance from `.env`), set `files: 'isolated'` in the `env` block:

```ts
env: {
  files: 'isolated',  // default: 'cascade'
},
```

- A staged build then reads **only** `.env.{stage}` â†’ `.env.{stage}.local` â€” nothing leaks in from `.env`/`.env.local`.
- Compile **hard-errors** if neither stage file exists. Note this includes the default stage: with isolation on, a plain `compile` (stage `prod`) requires a `.env.prod`.
- The build log lists which base-`.env` keys are not available to the isolated build.
- `WORKERS_API_KEY` resolution is stage-scoped too: if the stage file doesn't define one, the projectId-derived key is used â€” not the `.env` one.
- Real shell/CI env vars still win over files in both modes. `ai-worker dev` always uses the cascade (local runs, nothing ships).

## Controlling shipped env vars (`env` block)

By default, the CLI ships env vars into the deployed Lambda's `env.json` using a **prefix allowlist** (`OPENAI_*`, `ANTHROPIC_*`, `DATABASE_*`, `MONGODB_*`, `REDIS_*`, `UPSTASH_*`, `WORKER_*`, `WORKERS_*`, `WORKFLOW_*`, `REMOTION_*`, `QUEUE_JOB_*`) plus any key your worker code actually references via `process.env.KEY`.

To control this precisely, add an optional `env` block to your microfox config â€” the root of `microfox.json`, or `workflowSettings.deploymentConfig` in `microfox.config.ts` (the same object that carries `worker.externalDeps` / `worker.groups`):

```ts
env: {
  // 'all-detected' (default): prefix allowlist + keys detected from code â€” unchanged if omitted
  // 'explicit': ONLY the include list (+ platform keys) ships, nothing else
  mode: 'all-detected',

  // Extra keys to ship ('*' wildcard supported); shipped if present in the env file cascade
  include: ['SOME_SDK_KEY', 'MYAPP_*'],

  // Keys to never ship â€” wins over include and the prefix allowlist
  exclude: ['DEBUG_*', 'LOCAL_ONLY_TOKEN'],

  // Per-group overlays â€” only affect that group's env.json.
  // include/exclude lists APPEND to the project lists; `mode` OVERRIDES the
  // project mode for that one group (e.g. lock down a single group to an
  // explicit key list while the rest stay on the default allowlist).
  groups: {
    video: { include: ['REMOTION_LICENSE_KEY'] },
    core:  { exclude: ['HEAVY_FEATURE_FLAG'] },
    billing: { mode: 'explicit', include: ['STRIPE_KEY'] },
  },
},
```

**Rules:**

- **Always shipped, cannot be excluded**: `ENVIRONMENT`, `STAGE`, `NODE_ENV`, and `WORKERS_API_KEY` (when resolved).
- **`AWS_*` never ships**, regardless of config (reserved by Lambda).
- **Precedence per key**: platform keys â†’ `AWS_*` block â†’ `exclude` â†’ `include` â†’ mode (`all-detected` allowlist+detected, or `explicit` = nothing else).
- **Per-group `mode`** overrides the project mode for that group's env.json only; a group without `mode` inherits the project mode. Group `include`/`exclude` lists always append to the project lists.
- `include` patterns that match nothing in your env files print a warning at compile time; in `explicit` mode the CLI logs exactly which keys ship.
- **No `env` block = behavior is 100% unchanged** from previous versions.

## How `compile` Works

### 1. Worker Discovery

Scans for files under `--ai-path` (default `app/ai`):
- Workers: `**/*.worker.ts`
- Queues: `**/*.queue.ts`

For each worker file:
- Extracts `workerConfig` export
- Extracts `createWorker` call with `id`
- Validates worker configuration

### 2. Dependency Analysis

For each worker, analyzes:
- **Runtime dependencies**: Packages imported by worker code
- **Environment variables**: `process.env.*` usage
- **Worker-to-worker calls**: `ctx.dispatchWorker('worker-id', ...)` patterns

**Dependency detection:**
- Walks import graph from worker entry files
- Identifies external packages (non-builtin, non-local)
- Includes `@microfox/ai-worker` and `@aws-sdk/client-sqs` by default

**Environment variable detection:**
- Scans for `process.env.KEY` patterns
- Resolves values from the [env file cascade](#environment-files-env-cascade)
- Writes shipped keys to `env.json` per the [`env` block rules](#controlling-shipped-env-vars-env-block)

**Worker-to-worker detection:**
- Scans handler code for `ctx.dispatchWorker('worker-id', ...)`
- Extracts worker IDs from string literals
- Generates `WORKER_QUEUE_URL_<SANITIZED_ID>` env vars

### 3. Handler Generation

Generates Lambda handler files in `.serverless-workers/handlers/`:

```typescript
// .serverless-workers/handlers/worker-id.ts
import { createLambdaHandler } from '@microfox/ai-worker/handler';
import worker from '../../app/ai/workers/worker-id.worker';

export const handler = createLambdaHandler(
  worker.handler,
  worker.outputSchema
);
```

### 4. Queue Handler Generation

For queues, generates:
- **Queue starter handler**: Lambda triggered by schedule (if configured)
- **Queue step handlers**: Wrappers that chain workers sequentially

### 5. Serverless Configuration

Generates `serverless.yml` with:

- **Functions**: One Lambda per worker (and queue starter)
- **SQS Queues**: One queue per worker (with DLQ if configured); queue names carry the stage
- **Event Sources**: SQS triggers for worker functions, schedules for queue starters
- **Environment Variables**: From `env.json`
- **Layers**: If `layers` specified in `workerConfig`
- **IAM Roles**: Permissions for SQS, Lambda, and job store access

**Example generated config:**

```yaml
service: my-service-workers
frameworkVersion: '3'

provider:
  name: aws
  runtime: nodejs20.x
  region: us-east-1
  stage: prod
  environment:
    STAGE: ${self:provider.stage}
    AWS_REGION: ${self:provider.region}
    MONGODB_URI: ${env:MONGODB_URI}
    WORKER_QUEUE_URL_VIDEO_PROCESSOR: ${self:custom.queues.video-processor.url}
  iam:
    role:
      statements:
        - Effect: Allow
          Action:
            - sqs:SendMessage
            - sqs:ReceiveMessage
            - sqs:DeleteMessage
          Resource:
            - ${self:custom.queues.*.arn}

functions:
  video-processor:
    handler: handlers/video-processor.handler
    timeout: 900
    memorySize: 2048
    events:
      - sqs:
          arn: ${self:custom.queues.video-processor.arn}
          batchSize: 1
    environment:
      WORKER_ID: video-processor

resources:
  Resources:
    VideoProcessorQueue:
      Type: AWS::SQS::Queue
      Properties:
        QueueName: ${self:provider.stage}-video-processor
        VisibilityTimeout: 960
        MessageRetentionPeriod: 86400
        RedrivePolicy:
          deadLetterTargetArn: ${self:custom.queues.video-processor.dlqArn}
          maxReceiveCount: 3
```

### 6. Build Process

Uses `esbuild` to bundle handlers:

- **Entry points**: Generated handler files
- **External dependencies**: Excluded per `worker.externalDeps` / `worker.includeNodeModules` config
- **Output**: `.serverless-workers/dist/`
- **Format**: CommonJS (for Lambda compatibility)

### 7. Deploy Process (microfox CLI)

`microfox push` bundles the build as a ZIP and uploads it to the Microfox platform, which then:

- Runs `serverless deploy` server-side
- Creates/updates SQS queues, Lambda functions, and event source mappings
- Applies the environment variables from `env.json`
- Records the deployment (visible via `microfox status`/`logs` and the console)

## Configuration Files

### Env files

The CLI reads environment variables from the [stage-scoped cascade](#environment-files-env-cascade) (`.env` â†’ `.env.local` â†’ `.env.{stage}` â†’ `.env.{stage}.local`):

```bash
# Required (adapt to your job store)
MONGODB_URI=mongodb://...

# Recommended: shared secret for deployed worker endpoints
WORKERS_API_KEY=secret-key

# Anything your workers reference via process.env
OPENAI_API_KEY=sk-...
```

### `serverless.yml` (Generated)

The CLI generates `serverless.yml` in `.serverless-workers/`. You can inspect it, but it is overwritten on the next `compile`.

## Worker Configuration Extraction

The CLI extracts `workerConfig` from worker files:

```typescript
// âś… Good: Separate export
export const workerConfig: WorkerConfig = {
  timeout: 900,
  memorySize: 2048,
};

// âťŚ Bad: Inline in createWorker (deprecated)
export default createWorker({
  id: 'worker',
  workerConfig: { timeout: 900 }, // CLI won't extract this
});
```

**Supported config fields:**

- `timeout` - Lambda timeout (seconds, max 900)
- `memorySize` - Lambda memory (MB, 128-10240)
- `layers` - Lambda layer ARNs
- `schedule` - Schedule events (cron/rate)
- `group` - Deployment group (multi-group layout)
- `sqs.maxReceiveCount` - DLQ threshold
- `sqs.messageRetentionPeriod` - Queue retention (seconds)
- `sqs.visibilityTimeout` - Queue visibility timeout (seconds)

## Queue Configuration

For queue files (`*.queue.ts`), the CLI:

1. Extracts `defineWorkerQueue` calls
2. Generates queue starter Lambda (if `schedule` is set)
3. Generates queue step handlers
4. Sets up SQS triggers for chaining

**Queue file example:**

```typescript
import { defineWorkerQueue } from '@microfox/ai-worker/queue';

export default defineWorkerQueue({
  id: 'data-pipeline',
  steps: [
    { workerId: 'data-collector' },
    { workerId: 'data-processor', delaySeconds: 10 },
    { workerId: 'data-uploader' },
  ],
  schedule: 'cron(0 3 * * ? *)',
});
```

## Environment Variable Injection

The CLI automatically injects environment variables:

**From worker code:**
- Scans for `process.env.*` usage
- Adds detected keys to the Lambda environment (subject to the [`env` block rules](#controlling-shipped-env-vars-env-block))

**For worker-to-worker:**
- Detects `ctx.dispatchWorker('worker-id', ...)`
- Injects `WORKER_QUEUE_URL_<SANITIZED_ID>` into caller's Lambda

**Sanitization:**
- Replaces `-` with `_`
- Converts to uppercase
- Example: `video-processor` â†’ `WORKER_QUEUE_URL_VIDEO_PROCESSOR`

## Troubleshooting

### Build Failures

**Issue:** Handler build fails with import errors

**Solution:** Check that all dependencies are listed in `package.json`. The CLI bundles external packages, but local imports must be resolvable. For packages that should not be bundled, use `worker.externalDeps` in your microfox config.

### Deployment Failures

**Issue:** `microfox push` / `deploy` fails or the deployment errors out

**Solution:**
- Run `npx microfox logs latest` to see the platform-side build logs (`compiling`/`deploying` phases usually contain the root cause)
- Ensure you are logged in (`npx microfox login`) and your config has a valid `projectId`
- A 401 response means you are not authenticated â€” run `microfox login` (or set `MICROFOX_TOKEN` in CI)

### Missing Environment Variables

**Issue:** Lambda fails with missing env vars

**Solution:**
- Add variables to your env files (remember the cascade: `.env` â†’ `.env.local` â†’ `.env.{stage}` â†’ `.env.{stage}.local`)
- If the key doesn't match the default prefix allowlist and isn't referenced via `process.env.KEY` in worker code, add it to `env.include`
- Check for an `env.exclude` pattern accidentally matching the key
- Verify what shipped by inspecting the build's `env.json` in `.serverless-workers/`

### Worker-to-Worker Not Working

**Issue:** `ctx.dispatchWorker` fails with missing queue URL

**Solution:**
- Ensure callee worker is deployed
- Check that `WORKER_QUEUE_URL_<SANITIZED_ID>` is in caller's Lambda environment
- For cross-group calls, set env var manually (see [Worker Architecture](../advanced/worker-architecture#multi-group-deployments))

## Best Practices

1. **Develop against `ai-worker dev`** - Fast local iteration with the same routes as production, no AWS needed
2. **Export `workerConfig` separately** - Makes it easy for CLI to extract
3. **Use consistent worker IDs** - Matches filename (without `.worker.ts`)
4. **Group workers by feature** - Use subdirectories (`app/ai/workers/video/`) and deployment groups for heavy dependencies
5. **Add `.serverless-workers/` to `.gitignore`** - It is regenerated on every compile
6. **Use stages** - Deploy `staging` and `prod` as parallel stacks with `--stage`
7. **Keep secrets scoped** - Use the `env` block (`mode: 'explicit'` or `exclude`) so only intended keys ship
8. **Monitor deployments** - `microfox status latest` / `microfox logs latest` after each deploy

## Integration with CI/CD

**GitHub Actions example:**

```yaml

  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - uses: actions/setup-node@v3

          node-version: '20'

        env:
          MICROFOX_TOKEN: ${{ secrets.MICROFOX_TOKEN }}
          MICROFOX_PROJECT_ID: ${{ secrets.MICROFOX_PROJECT_ID }}
```

No AWS credentials are needed in CI â€” the Microfox platform performs the AWS deployment. `MICROFOX_TOKEN` replaces interactive `microfox login`; the `STAGE` env var can replace `--stage`.
