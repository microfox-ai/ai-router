## Overview

`@microfox/ai-worker` provides a runtime for executing background jobs on AWS Lambda (triggered via SQS queues) or locally in development. It enables long-running tasks, worker-to-worker communication, and durable job state management through MongoDB or Redis.

## Core Types

### `createWorker<INPUT_SCHEMA, OUTPUT>(config)`

Creates a worker agent that can be dispatched to SQS/Lambda.

```typescript
export function createWorker<INPUT_SCHEMA extends ZodType<any>, OUTPUT>(
  config: WorkerAgentConfig<INPUT_SCHEMA, OUTPUT>
): WorkerAgent<INPUT_SCHEMA, OUTPUT>
```

**Parameters:**

- `config.id: string` - Unique worker identifier
- `config.inputSchema: INPUT_SCHEMA` - Zod schema for input validation
- `config.outputSchema: ZodType<OUTPUT>` - Zod schema for output validation
- `config.handler: WorkerHandler<z.infer<INPUT_SCHEMA>, OUTPUT>` - Handler function
- `config.retry?: SmartRetryConfig` - Smart retry configuration (in-process retries on matching errors). See [Smart Retry](#smart-retry).
- `config.workerConfig?: WorkerConfig` - **Deprecated**: Prefer exporting `workerConfig` separately

**Returns:** `WorkerAgent<INPUT_SCHEMA, OUTPUT>` with a `dispatch()` method

**Example:**

```typescript
import { createWorker } from '@microfox/ai-worker';
import { z } from 'zod';

const InputSchema = z.object({
  url: z.string().url(),
  timeout: z.number().optional().default(5000),
});

const OutputSchema = z.object({
  processedUrl: z.string(),
  duration: z.number(),
});

export default createWorker<typeof InputSchema, z.infer<typeof OutputSchema>>({
  id: 'video-processor',
  inputSchema: InputSchema,
  outputSchema: OutputSchema,
  handler: async ({ input, ctx }) => {
    await ctx.jobStore?.update({ status: 'running' });
    const result = await processVideo(input.url);
    return { processedUrl: result.url, duration: result.duration };
  },
});
```

### `WorkerAgent<INPUT_SCHEMA, OUTPUT>`

The worker agent object returned by `createWorker`.

```typescript
export interface WorkerAgent<INPUT_SCHEMA extends ZodType<any>, OUTPUT> {
  id: string;
  dispatch: (
    input: z.input<INPUT_SCHEMA>,
    options: DispatchOptions
  ) => Promise<DispatchResult>;
  handler: WorkerHandler<z.infer<INPUT_SCHEMA>, OUTPUT>;
  inputSchema: INPUT_SCHEMA;
  outputSchema: ZodType<OUTPUT>;
  workerConfig?: WorkerConfig;
  /** Smart retry config set via createWorker({ retry }). */
  retry?: SmartRetryConfig;
}
```

### `WorkerHandler<INPUT, OUTPUT>`

The handler function signature for worker execution.

```typescript
export type WorkerHandler<INPUT, OUTPUT> = (
  params: WorkerHandlerParams<INPUT, OUTPUT>
) => Promise<OUTPUT>;
```

**Handler Parameters:**

```typescript
export interface WorkerHandlerParams<INPUT, OUTPUT> {
  input: INPUT; // Parsed input (defaults applied)
  ctx: {
    jobId: string;
    workerId: string;
    requestId?: string;
    /** Propagated from DispatchOptions.userId; attach from getClientId() in your API route. */
    userId?: string;
    jobStore?: JobStore;
    /** Logger with prefixed levels: [INFO] [workerId] [jobId] message */
    logger: {
      info(message: string, data?: Record<string, unknown>): void;
      warn(message: string, data?: Record<string, unknown>): void;
      error(message: string, data?: Record<string, unknown>): void;
      debug(message: string, data?: Record<string, unknown>): void;
    };
    dispatchWorker: (
      workerId: string,
      input: unknown,
      options?: DispatchWorkerOptions
    ) => Promise<{ jobId: string; messageId?: string; output?: unknown }>;
    /**
     * Report LLM token usage after each call. Accumulates across all calls.
     * Throws TokenBudgetExceededError if maxTokens (from DispatchOptions) is exceeded.
     * Also persists usage to the job store (metadata.tokenUsage) for observability.
     */
    reportTokenUsage: (usage: { inputTokens: number; outputTokens: number }) => Promise<void>;
    /**
     * Get the current token usage and remaining budget for this job.
     * Returns { used, budget: null, remaining: null } when no maxTokens was set.
     */
    getTokenBudget: () => { used: number; budget: number | null; remaining: number | null };
    /**
     * Populated on retry attempts (attempt >= 2). Undefined on the first attempt.
     * Use to self-correct: inject lastError.message into the next LLM prompt.
     */
    retryContext?: RetryContext;
    [key: string]: any;
  };
}
```

## Worker Configuration

### `WorkerConfig`

Lambda deployment configuration for a worker. **Best Practice**: Export this as a separate const from your worker file. The CLI reads it when generating `serverless.yml`; you do not pass it into `createWorker()`.

```typescript
export interface WorkerConfig {
  timeout?: number; // Lambda timeout in seconds (max 900)
  memorySize?: number; // Lambda memory in MB (128-10240)
  layers?: string[]; // Lambda layer ARNs
  schedule?: ScheduleConfig; // Schedule events (cron/rate)
  group?: string; // Multi-group: deploy this worker to a different stack (max 12 chars; cannot be 'core'; omit for 'default')
  sqs?: {
    maxReceiveCount?: number; // DLQ threshold (default: 3)
    messageRetentionPeriod?: number; // Seconds (max 1209600)
    visibilityTimeout?: number; // Seconds (default: timeout + 60)
    deadLetterMessageRetentionPeriod?: number; // DLQ retention
  };
}
```

**Multi-group:** Set `group: 'scraper'` (or any name except `core`) to deploy this worker to a separate Serverless stack. One `WORKER_BASE_URL` (core stack) is used to trigger all workers. See [Worker Architecture – Multi-group deployments](../advanced/worker-architecture#multi-group-deployments) for details.

**Example:**

```typescript
import { type WorkerConfig } from '@microfox/ai-worker';

export const workerConfig: WorkerConfig = {
  timeout: 900, // 15 minutes
  memorySize: 2048, // 2GB
  layers: ['arn:aws:lambda:${aws:region}:${aws:accountId}:layer:ffmpeg:1'],
  schedule: 'rate(2 hours)',
  sqs: {
    maxReceiveCount: 3,
    messageRetentionPeriod: 86400, // 1 day
  },
};
```

### `ScheduleConfig`

Schedule event configuration for periodic execution.

```typescript
export type ScheduleConfig = 
  | string // Simple rate/cron: 'rate(2 hours)' or 'cron(0 12 * * ? *)'
  | ScheduleEventConfig
  | (string | ScheduleEventConfig)[]; // Multiple schedules

export interface ScheduleEventConfig {
  rate: string | string[]; // Rate or cron expression(s)
  enabled?: boolean; // Default: true
  input?: Record<string, any>; // Input payload
  inputPath?: string; // JSONPath to select event data
  inputTransformer?: {
    inputPathsMap?: Record<string, string>;
    inputTemplate?: string;
  };
  name?: string; // Schedule name
  description?: string;
  method?: 'eventBus' | 'scheduler'; // Default: 'eventBus'
  timezone?: string; // Only for 'scheduler' method
}
```

**Examples:**

```typescript
// Simple rate
schedule: 'rate(2 hours)'

// Multiple schedules
schedule: [
  'rate(2 hours)',
  { rate: 'cron(0 12 * * ? *)', enabled: true, input: { key: 'value' } }
]

// Scheduler method with timezone
schedule: {
  method: 'scheduler',
  rate: 'cron(0 0/4 ? * MON-FRI *)',
  timezone: 'America/New_York',
  input: { key1: 'value1' }
}
```

## Dispatch API

### `dispatch(input, options)`

Dispatches a job to the background worker. In local/auto mode (development), runs the handler immediately in-process. In remote mode (production), sends a message to SQS which triggers a Lambda function.

```typescript
async dispatch(
  input: z.input<INPUT_SCHEMA>,
  options: DispatchOptions
): Promise<DispatchResult>
```

**Parameters:**

- `input: z.input<INPUT_SCHEMA>` - Input data (validated against `inputSchema`)
- `options: DispatchOptions` - Dispatch options

**DispatchOptions:**

```typescript
export interface DispatchOptions {
  webhookUrl?: string; // Callback URL when job completes
  mode?: 'auto' | 'local' | 'remote'; // Execution mode
  jobId?: string; // Optional job ID (default: generated)
  metadata?: Record<string, any>; // Metadata attached to job
  userId?: string; // Optional user ID for audit logging (attach from getClientId())
  registry?: WorkerQueueRegistry; // For dispatchQueue
  onCreateQueueJob?: (params: {
    queueJobId: string;
    queueId: string;
    firstStep: { workerId: string; workerJobId: string };
    metadata?: Record<string, unknown>;
  }) => Promise<void>;
  /**
   * Maximum total tokens (input + output) allowed for this job.
   * The worker must call ctx.reportTokenUsage() after each LLM call.
   * Throws TokenBudgetExceededError (never retried) when the limit is reached.
   * Token usage is stored in metadata.tokenUsage for observability.
   */
  maxTokens?: number;
}
```

**Returns:**

```typescript
export interface DispatchResult {
  messageId: string;
  status: 'queued';
  jobId: string;
}
```

**Example:**

```typescript
const result = await worker.dispatch(
  { url: 'https://example.com/video.mp4' },
  {
    webhookUrl: 'https://myapp.com/api/callback',
    mode: 'auto',
    metadata: { userId: '123' },
  }
);
// Returns: { messageId: '...', status: 'queued', jobId: '...' }
```

## Worker-to-Worker Communication

### `ctx.dispatchWorker(workerId, input, options?)`

Dispatch another worker from within a worker handler.

```typescript
dispatchWorker: (
  workerId: string,
  input: unknown,
  options?: DispatchWorkerOptions
) => Promise<{ jobId: string; messageId?: string; output?: unknown }>
```

**DispatchWorkerOptions:**

```typescript
export interface DispatchWorkerOptions {
  webhookUrl?: string;
  metadata?: Record<string, any>;
  jobId?: string;
  await?: boolean; // Poll until completion
  pollIntervalMs?: number; // Default: 2000
  pollTimeoutMs?: number; // Default: 15 minutes
  delaySeconds?: number; // 0-900, fire-and-forget only
}
```

**Examples:**

```typescript
handler: async ({ ctx }) => {
  // Fire-and-forget
  await ctx.dispatchWorker('other-worker', { key: 'value' });
  
  // Fire-and-forget with delay (0-900 seconds)
  await ctx.dispatchWorker('other-worker', { key: 'value' }, {
    delaySeconds: 60
  });
  
  // Await completion
  const result = await ctx.dispatchWorker('other-worker', { key: 'value' }, {
    await: true,
    pollIntervalMs: 3000,
    pollTimeoutMs: 600000
  });
  console.log(result.output); // Child worker output
  
  // Append to internalJobs automatically
  // Parent job's internalJobs will include: [{ jobId: '...', workerId: 'other-worker' }]
}
```

**Deployment:** The CLI scans for `ctx.dispatchWorker('worker-id', ...)` and injects `WORKER_QUEUE_URL_<SANITIZED_ID>` into that Lambda's environment. Same-group workers get queue URLs and IAM automatically; for cross-group dispatch (calling a worker in another stack), see [Worker Architecture – Multi-group](../advanced/worker-architecture#multi-group-deployments).

**Local:** In development, uses HTTP trigger (`WORKER_BASE_URL`) when queue URL env is not set.

## Queue API

### `defineWorkerQueue(config)`

Defines a worker queue (sequential chain of workers).

```typescript
export function defineWorkerQueue<T extends WorkerQueueConfig>(config: T): T
```

**WorkerQueueConfig:**

```typescript
export interface WorkerQueueConfig {
  id: string; // Queue identifier
  steps: WorkerQueueStep[]; // Ordered workers
  schedule?: string | { rate: string; enabled?: boolean; input?: Record<string, any> };
}

export interface WorkerQueueStep {
  workerId: string;
  delaySeconds?: number; // 0–900, SQS DelaySeconds

  /**
   * Called when the queue advances to this step after the previous step completes.
   * Built-in shortcuts: 'passthrough' | 'continueFromPrevious'
   * When omitted, the previous output is passed as-is.
   */
  chain?: ((ctx: ChainContext) => unknown) | 'passthrough' | 'continueFromPrevious';

  /**
   * Called when this step resumes after a human approves a HITL pause.
   * Returns the final domain input for the worker handler.
   * Default: shallow merge of pendingInput + reviewerInput.
   */
  resume?: (ctx: HitlResumeContext<any>) => unknown;

  /**
   * When true, queue execution pauses before dispatching this step.
   * A human must POST to /queues/:queueId/approve to continue.
   */
  requiresApproval?: boolean;

  /**
   * Optional HITL UI metadata consumed by the app UI (not by the worker runtime).
   * Use defineHitlConfig() for type-safe authoring.
   */
  hitl?: HitlStepConfig;

  /**
   * Smart retry config for this step. Overrides worker-level retry for this step only.
   * Retries run in-process (same Lambda invocation). ctx.retryContext is populated
   * on each retry so the handler can self-correct (e.g. inject error into prompt).
   * TokenBudgetExceededError is never retried.
   *
   * @example { maxAttempts: 3, on: ['rate-limit', 'json-parse'] }
   */
  retry?: SmartRetryConfig;

  /**
   * When defined, evaluated after each run to decide whether to re-run this step
   * (loop) instead of advancing to the next step.
   */
  loop?: {
    shouldContinue: (ctx: LoopContext) => boolean | Promise<boolean>;
    maxIterations?: number; // Default: 50
  };
}
```

**Chain and Resume Context Types:**

```typescript
export interface QueueStepOutput {
  stepIndex: number;
  workerId: string;
  output: unknown;
}

export interface ChainContext {
  initialInput: unknown;          // Input passed to dispatchQueue
  previousOutputs: QueueStepOutput[];  // All completed step outputs
}

export interface HitlResumeContext<T = unknown> {
  initialInput: unknown;
  previousOutputs: QueueStepOutput[];
  reviewerInput: T;               // Payload submitted via approve endpoint
  pendingInput: Record<string, unknown>; // What chain() produced before pause
}

export interface LoopContext {
  output: unknown;                // Current iteration output
  stepIndex: number;              // Queue definition step index
  iterationCount: number;         // 0-based iteration count
  initialInput: unknown;
  previousOutputs: QueueStepOutput[];
}
```

**Example:**

```typescript
import { defineWorkerQueue } from '@microfox/ai-worker/queue';

export default defineWorkerQueue({
  id: 'data-pipeline',
  steps: [
    { workerId: 'data-collector' },
    {
      workerId: 'data-processor',
      delaySeconds: 10,
      chain: (ctx) => ({
        data: ctx.previousOutputs.at(-1)?.output,
        source: (ctx.initialInput as any).source,
      }),
    },
    {
      workerId: 'review-step',
      requiresApproval: true,
      chain: 'passthrough',
      resume: (ctx) => ({ ...ctx.pendingInput, approved: ctx.reviewerInput }),
      hitl: defineHitlConfig({
        taskKey: 'review-data',
        ui: { type: 'schema-form', title: 'Review processed data' },
      }),
    },
  ],
  schedule: 'cron(0 3 * * ? *)', // Daily at 03:00 UTC
});
```

### `defineHitlConfig(config)`

Type-safe helper for authoring HITL step metadata. The config is attached to `WorkerQueueStep.hitl` and used by the app UI — the worker runtime only reads `requiresApproval`.

```typescript
export function defineHitlConfig<T extends HitlStepConfig>(config: T): T
```

**HitlStepConfig:**

```typescript
export type HitlStepConfig = {
  taskKey: string;                     // Unique key for this HITL step
  timeoutSeconds?: number;             // Auto-reject/approve after this duration
  onTimeout?: 'reject' | 'auto-approve';
  assignees?: string[];                // User IDs to notify
  ui: HitlUiSpec;                      // Rendering hint for the frontend
  inputSchema?: ZodType;              // Reviewer form schema
};

export type HitlUiSpec =
  | {
      type: 'custom';
      viewId: string;          // Looked up in the app's HITL view registry
      title?: string;
      sections?: Array<Record<string, unknown>>;
      [key: string]: unknown;
    }
  | {
      type: 'schema-form';     // Auto-render a form; falls back to JSON textarea
      title?: string;
      description?: string;
    };
```

**Example:**

```typescript
import { defineHitlConfig } from '@microfox/ai-worker/hitlConfig';
import { z } from 'zod';

const hitl = defineHitlConfig({
  taskKey: 'review-step',
  timeoutSeconds: 3600,
  onTimeout: 'reject',
  ui: { type: 'schema-form', title: 'Review the output', description: 'Approve or reject the data' },
  inputSchema: z.object({
    approved: z.boolean(),
    comment: z.string().optional(),
  }),
});
```

For custom views (e.g. a specialized UI component), use `type: 'custom'` with a `viewId` registered in `components/hitl/registry.ts`:

```typescript

  taskKey: 'calculator-hitl',
  ui: {
    type: 'custom',
    viewId: 'calculator-hitl-v1',
    title: 'Calculator review',
    sections: ['calculator-keypad', 'hint'],
  },
  inputSchema: calculatorHitlInputSchema,
});
```

See [HITL Queues](../advanced/hitl-queues) for the full HITL workflow and frontend integration guide.

### `dispatchQueue` — HITL approve endpoint

When a queue has a step with `requiresApproval: true`, execution pauses at that step. The frontend calls the approve endpoint to resume:

```
POST /api/workflows/queues/:queueId/approve
```

**Body:**

```json
{
  "queueJobId": "job-123",
  "stepIndex": 1,
  "decision": "approve",
  "input": { "approved": true, "comment": "Looks good" },
  "reviewerId": "user-456",
  "comment": "Optional comment"
}
```

- `decision`: `"approve"` (default) or `"reject"` — reject marks the step as failed
- `input`: reviewer form payload, passed to the step's `resume()` function as `reviewerInput`
- `stepIndex`: the array step index (auto-detected if omitted)

**Response:**

```json
{ "ok": true, "decision": "approve", "queueJobId": "job-123", "stepIndex": 1, "workerId": "review-step", "status": "running" }
```

The runtime writes `status: 'running'` to the store **before** dispatching the worker (race-condition guard), then calls `resume()` inside the Lambda to merge `pendingInput` and `reviewerInput` before the handler runs.

## Environment Variables

### Next.js Application

**For triggering workers (remote, server-side only):**
- `WORKER_BASE_URL` - Base URL of deployed workers service
- `WORKERS_TRIGGER_API_KEY` - Optional API key for trigger authentication
- `WORKERS_CONFIG_API_KEY` - Optional API key for config API authentication

**For job store:**
- `WORKER_DATABASE_TYPE` - `'mongodb'` or `'upstash-redis'` (default: `'upstash-redis'`)
- **MongoDB**: `MONGODB_URI` or `DATABASE_MONGODB_URI`, `MONGODB_DB` or `DATABASE_MONGODB_DB`, `MONGODB_WORKER_JOBS_COLLECTION`
- **Redis**: `WORKER_UPSTASH_REDIS_REST_URL` (or `UPSTASH_REDIS_REST_URL`), `WORKER_UPSTASH_REDIS_REST_TOKEN` (or `UPSTASH_REDIS_REST_TOKEN`), `WORKER_UPSTASH_REDIS_JOBS_PREFIX` (default: `worker:jobs:`), `WORKER_JOBS_TTL_SECONDS`

**Local development:**
- `WORKERS_LOCAL_MODE` - Set to `'false'` to disable local execution (default: auto-detects from `NODE_ENV`)
- `WORKER_JOB_STORE_MODULE_PATH` - Custom job store module path (for direct DB access in local mode)
- `WORKER_JOB_STORE_URL` - Job store HTTP URL (fallback when direct DB access unavailable)

### AWS Lambda (deployed workers)

**Required:**
- `AWS_REGION` - AWS region for SQS/Lambda
- `STAGE` - Deployment stage (dev/stage/prod)
- `MONGODB_URI` or `DATABASE_MONGODB_URI` - For MongoDB job store
- **OR** `WORKER_UPSTASH_REDIS_REST_URL` and `WORKER_UPSTASH_REDIS_REST_TOKEN` - For Redis job store
- `WORKER_QUEUE_URL_<SANITIZED_ID>` - Auto-injected by CLI for worker-to-worker (e.g., `WORKER_QUEUE_URL_VIDEO_PROCESSOR`)

**Optional:**
- `DEBUG_WORKER_QUEUES` - Set to `'1'` to enable queue debugging logs
- `WORKFLOW_APP_BASE_URL` or `QUEUE_JOB_API_URL` - Base URL of your app (for queue job step updates from Lambda)

## Smart Retry

Workers support **in-process smart retry**: when the handler throws a matching error, the runtime retries it within the same Lambda invocation (no SQS re-delivery, job stays `running`). On each retry, `ctx.retryContext` is populated so the handler can self-correct.

### `SmartRetryConfig`

```typescript
export interface SmartRetryConfig {
  /** Maximum total attempts including the first (default: 3). */
  maxAttempts?: number;
  /** Error patterns that trigger a retry. Non-matching errors fail immediately. */
  on: RetryPattern[];
}

export type RetryPattern = BuiltInRetryPattern | CustomRetryPattern;

export type BuiltInRetryPattern =
  | 'rate-limit'   // HTTP 429, RateLimitError — delays: 10s, 20s, 30s…
  | 'json-parse'   // SyntaxError, ZodError — immediate, injects ctx.retryContext
  | 'overloaded'   // HTTP 529, "model is busy" — delays: 15s, 30s…
  | 'server-error'; // HTTP 5xx, "internal server error" — delays: 5s, 10s…

export interface CustomRetryPattern {
  /** Regex test against error.message, or a predicate receiving the full error. */
  match: RegExp | ((err: Error & Record<string, any>) => boolean);
  /** Delay before the retry. A function receives the retry attempt number. */
  delayMs?: number | ((attempt: number) => number);
  /** When true, populates ctx.retryContext.lastError so the handler can self-correct. */
  injectContext?: boolean;
}
```

### `RetryContext`

```typescript
export interface RetryContext {
  /** Current attempt number (1-indexed). 2 = first retry. */
  attempt: number;
  /** Total max attempts configured. */
  maxAttempts: number;
  /** Error from the previous attempt — use to self-correct (inject into prompt). */
  lastError: {
    message: string;
    name: string;
    stack?: string;
    /** HTTP status code or error code if present. */
    code?: string | number;
  };
}
```

### Setting retry on a worker

```typescript
import { createWorker } from '@microfox/ai-worker';

export default createWorker({
  id: 'llm-worker',
  inputSchema, outputSchema,
  retry: {
    maxAttempts: 3,
    on: ['rate-limit', 'json-parse'],
  },
  handler: async ({ input, ctx }) => {
    // ctx.retryContext is undefined on first attempt, populated on retries
    const errorHint = ctx.retryContext
      ? `\n\nPrevious attempt failed: "${ctx.retryContext.lastError.message}". Correct your output.`
      : '';

    const response = await anthropic.messages.create({
      messages: [{ role: 'user', content: input.prompt + errorHint }],
    });

    // SyntaxError triggers 'json-parse' retry with ctx.retryContext injected
    return JSON.parse(response.content[0].text);
  },
});
```

### Setting retry on a queue step

A step's `retry` overrides the worker-level retry for that step only:

```typescript
defineWorkerQueue({
  id: 'pipeline',
  steps: [
    { workerId: 'ingest' },
    {
      workerId: 'llm-extractor',
      // Override: up to 5 attempts for this step, with progressive delay
      retry: {
        maxAttempts: 5,
        on: [
          'rate-limit',
          'json-parse',
          { match: /quota exceeded/i, delayMs: (attempt) => attempt * 5000 },
        ],
      },
    },
  ],
});
```

### Built-in pattern delays

| Pattern | Delay formula |
|---------|--------------|
| `'rate-limit'` | `attempt * 10_000` ms (10s, 20s, 30s…) |
| `'json-parse'` | `0` ms (immediate; context injected) |
| `'overloaded'` | `attempt * 15_000` ms (15s, 30s…) |
| `'server-error'` | `attempt * 5_000` ms (5s, 10s…) |

> **Note**: `TokenBudgetExceededError` is **never retried**, even if matching patterns are configured.

## Token Budget

Pass `maxTokens` in `DispatchOptions` to cap total LLM token usage for a job. The worker calls `ctx.reportTokenUsage()` after each LLM call; when the accumulated total exceeds the budget, `TokenBudgetExceededError` is thrown and the job fails immediately (never retried).

### `TokenBudgetExceededError`

```typescript
export class TokenBudgetExceededError extends Error {
  readonly used: number;   // Total tokens used when limit was hit
  readonly budget: number; // The configured maxTokens limit
  // name: 'TokenBudgetExceededError'
}
```

### `TokenUsage` / `TokenBudgetState`

```typescript
export interface TokenUsage {
  inputTokens: number;
  outputTokens: number;
}

export interface TokenBudgetState {
  inputTokens: number;
  outputTokens: number;
  /** null when no budget was configured. */
  budget: number | null;
}
```

### Using the token budget

**Dispatching with a budget:**

```typescript
import { myWorker } from './workers/my-worker';

// Set a budget of 50,000 total tokens for this job
const result = await myWorker.dispatch(input, {
  maxTokens: 50_000,
  metadata: { source: 'api' },
});
```

**Reporting usage in the handler:**

```typescript
handler: async ({ input, ctx }) => {
  for (let i = 0; i < input.steps; i++) {
    const response = await anthropic.messages.create({ /* ... */ });

    // Report usage — throws TokenBudgetExceededError if over budget
    await ctx.reportTokenUsage({
      inputTokens: response.usage.input_tokens,
      outputTokens: response.usage.output_tokens,
    });

    // Check remaining budget before proceeding
    const { used, budget, remaining } = ctx.getTokenBudget();
    ctx.logger.info(`Step ${i + 1} complete`, { used, remaining });

    if (remaining !== null && remaining < 2000) {
      ctx.logger.warn('Low token budget — stopping early', { remaining });
      return partialResult(i);
    }
  }

  return finalResult();
},
```

**Token usage is stored in the job record:**

After each `reportTokenUsage()` call, the runtime writes the accumulated state to `metadata.tokenUsage`:

```json
{
  "jobId": "job-xxx",
  "status": "completed",
  "metadata": {
    "tokenUsage": {
      "inputTokens": 12400,
      "outputTokens": 3600,
      "budget": 50000
    }
  }
}
```

### Without a budget

When `maxTokens` is not set, `ctx.reportTokenUsage()` still accumulates and stores usage, but `TokenBudgetExceededError` is never thrown. `ctx.getTokenBudget()` returns `{ used, budget: null, remaining: null }`.

## Execution Modes

The `dispatch()` method supports three execution modes controlled by the `mode` option:

### Auto Mode (default)

`mode: 'auto'` (or omitted) automatically chooses based on environment:

- **Development** (`NODE_ENV === 'development'` and `WORKERS_LOCAL_MODE !== 'false'`): Runs handlers immediately in-process (local mode)
- **Production** (otherwise): Sends messages to SQS (remote mode)

### Local Mode

`mode: 'local'` forces in-process execution even in production:

- Handlers execute synchronously in the same Node.js process
- Job store uses direct DB access (if `WORKER_JOB_STORE_MODULE_PATH` is set) or HTTP fallback
- Webhooks are called immediately after completion
- Worker-to-worker uses HTTP trigger (`WORKER_BASE_URL`)

Perfect for testing and development.

### Remote Mode

`mode: 'remote'` forces SQS/Lambda dispatch even in development:

- Handlers execute on AWS Lambda (triggered by SQS)
- Job store uses MongoDB or Redis directly (from Lambda environment)
- Webhooks are called from Lambda after completion
- Worker-to-worker uses SQS queues (via `WORKER_QUEUE_URL_*` env vars)

Use this for production deployments or when testing Lambda behavior locally.

## Error Handling

Workers should handle errors gracefully and update job status:

```typescript
handler: async ({ input, ctx }) => {
  try {
    await ctx.jobStore?.update({ status: 'running' });
    const result = await processData(input);
    await ctx.jobStore?.update({ status: 'completed', output: result });
    return result;
  } catch (error: any) {
    await ctx.jobStore?.update({
      status: 'failed',
      error: {
        message: error.message || 'Unknown error',
        stack: error.stack,
        name: error.name || 'Error',
      },
    });
    throw error; // Re-throw to trigger Lambda error handling
  }
}
```

## Type Safety

The `createWorker` function uses TypeScript generics to ensure type safety:

- `INPUT_SCHEMA` - Zod schema type (e.g., `typeof InputSchema`)
- `z.input<INPUT_SCHEMA>` - Pre-parse input type (preserves optional fields)
- `z.infer<INPUT_SCHEMA>` - Parsed input type (defaults applied)
- `OUTPUT` - Handler return type

This ensures:
- `dispatch()` accepts pre-parse input (optional fields preserved)
- `handler` receives parsed input (defaults applied)
- Output is validated against `outputSchema`
